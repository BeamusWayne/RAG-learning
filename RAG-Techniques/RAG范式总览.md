# RAG 技术范式总览

这份文档解释的不是 LangChain、向量数据库、Embedding 模型这类“组件技术栈”，而是 **RAG 的不同架构范式**。

你可以把 RAG 理解成一个“开卷考试系统”：

1. 用户提问。
2. 系统去资料库里找相关材料。
3. 大模型阅读材料。
4. 大模型基于材料回答。

不同 RAG 范式的区别，主要在于：

- 怎么改写问题？
- 怎么找资料？
- 找到的资料靠不靠谱？
- 是否需要多轮检索？
- 是否需要知识图谱？
- 是否让 Agent 自己规划检索过程？
- 如何减少幻觉？

---

## 先用一张表建立全局印象

| 范式 | 一句话理解 | 主要解决的问题 | 适合场景 | 主要代价 |
|---|---|---|---|---|
| Naive RAG | 最基础的“检索 + 生成” | 让模型基于外部知识回答 | 入门 Demo、小型知识库 | 容易检索不准 |
| Advanced RAG | 给基础 RAG 加查询改写、混合检索、重排等增强 | 提升召回和答案质量 | 大多数生产级问答系统 | 调参和评测复杂 |
| Modular RAG | 把 RAG 拆成可组合模块 | 复杂业务可扩展 | 企业系统、多业务路由 | 链路长、排错难 |
| HyDE | 先生成“假想答案”，再拿它去检索 | 用户问题太短或太模糊 | 模糊查询、语义召回 | 假想答案可能带偏检索 |
| RAG-Fusion | 把一个问题改写成多个问题并行检索 | 单一查询召回不全 | 问法多样、信息分散 | 检索次数更多 |
| Hybrid RAG | 同时用向量、关键词、结构化检索 | 纯向量检索不稳定 | 代码、法规、产品文档 | 排序融合复杂 |
| Rerank RAG | 先粗召回，再用重排模型精排 | Top-K 里噪声太多 | 高质量问答 | 增加延迟和成本 |
| Parent-Child RAG | 小块负责匹配，大块负责回答 | 精确召回和上下文完整难兼顾 | 长文档、手册、论文 | 索引结构更复杂 |
| Hierarchical RAG | 先找文档/章节，再找段落 | 文档层级重要 | 书籍、法律、技术文档 | 构建层级索引麻烦 |
| RAPTOR | 把文档递归总结成摘要树 | 全局理解和局部细节兼顾 | 大规模长文档集合 | 摘要构建成本高 |
| Multi-hop RAG | 多步检索、多步推理 | 一个问题需要多个证据 | 复杂推理、跨文档问题 | 错误会逐步累积 |
| Self-RAG | 模型自己判断是否检索、证据是否足够 | 固定检索策略太死 | 需要自适应检索的任务 | 依赖模型自评能力 |
| CRAG | 先评估检索结果，不可靠就纠正 | 检索错了还强行回答 | 高可靠问答、开放域问答 | 多一步质量判断 |
| Adaptive RAG | 根据问题难度选择不同 RAG 策略 | 简单问题和复杂问题用同一套流程太浪费 | 成本敏感的生产系统 | 路由器要可靠 |
| Agentic RAG | 让 Agent 自己规划、检索、观察、再行动 | 复杂任务不是一次检索能解决 | 研究助手、自动分析、多工具系统 | 不稳定、慢、贵 |
| GraphRAG | 抽取实体关系，构建知识图谱辅助检索 | 关系推理和全局总结弱 | 企业知识、人物关系、事件网络 | 构图和维护成本高 |
| Long-context RAG | 把大量候选内容直接塞进长上下文模型 | 召回不准时希望模型自己筛 | 长上下文模型可用的场景 | 贵、慢、不一定读得准 |
| Multimodal RAG | 检索文本、图片、表格、音频、视频 | 知识不只存在于纯文本 | 论文、财报、医疗、工业文档 | 解析和评测难 |
| Cache-Augmented RAG | 缓存问题、检索结果或答案 | 降低延迟和成本 | 高频问题、客服系统 | 缓存过期会返回旧答案 |

## 1. Naive RAG：基础 RAG

### 它是什么

Naive RAG 是最经典、最容易理解的 RAG 流程：

```text
用户问题 -> 向量检索 Top-K 文档片段 -> 拼进 Prompt -> LLM 生成答案
```

比如用户问：

> 公司年假政策是什么？

系统会去知识库里找和“年假政策”最相似的几个文档片段，然后让模型基于这些片段回答。

### 它解决什么问题

大模型自身不知道你的私有知识库，也不知道最新资料。Naive RAG 通过外部检索，让模型临时“看到”相关材料。

### 优点

- 最容易实现，适合入门。
- 成本低，链路短，容易调试。
- 对 FAQ、简单文档问答、小型知识库很有效。

### 缺点

- 用户问题如果表达模糊，检索可能找错。
- 只看 Top-K 片段，容易漏掉关键证据。
- 对多跳推理、跨文档归纳、冲突信息处理能力弱。
- 如果检索结果本身错误，模型往往会基于错误材料认真回答。

### 适合场景

- RAG 入门项目。
- 文档结构简单的知识库问答。
- 公司制度、产品说明、课程资料这类相对明确的问题。

### 初学者记忆法

Naive RAG 就是：“先搜几段，再让模型照着回答。”

## 2. Advanced RAG：高级 RAG

### 它是什么

Advanced RAG 不是单一论文方法，而是对基础 RAG 的一组常见增强。它通常会加入：

- Query Rewrite：把用户问题改写得更适合检索。
- Hybrid Search：向量检索 + 关键词检索。
- Rerank：对初步召回结果重新排序。
- Metadata Filter：按时间、来源、权限、类型过滤。
- Context Compression：压缩上下文，只保留关键句。

典型流程：

```text
用户问题
-> 问题改写
-> 多路检索
-> 重排序
-> 上下文压缩
-> LLM 回答
```

### 它解决什么问题

基础 RAG 太“粗糙”。Advanced RAG 的目标是让检索更准、上下文更干净、回答更可靠。

### 优点

- 生产环境中最常见，性价比高。
- 能明显提升检索质量和答案质量。
- 可以逐步增强，不必一次性重构整个系统。

### 缺点

- 每加一个模块，就多一个需要调参和评测的地方。
- 延迟和成本会上升。
- 如果没有评测集，很容易不知道到底是哪一步出了问题。

### 适合场景

- 准备从 Demo 走向生产的 RAG 系统。
- 文档量较大、问题类型较复杂的知识库。
- 对准确率有要求，但还不需要 Agent 或图谱的系统。

### 初学者记忆法

Advanced RAG 就是：“基础 RAG 加强版，把检索前后都优化一遍。”

---

## 3. Modular RAG：模块化 RAG

### 它是什么

Modular RAG 把 RAG 拆成多个可以独立替换的模块：

- 查询理解模块。
- 查询改写模块。
- 检索路由模块。
- 多路检索模块。
- 重排模块。
- 上下文压缩模块。
- 答案生成模块。
- 答案验证模块。

它强调的不是某个具体算法，而是系统架构上的“可组合”。

### 它解决什么问题

当业务越来越复杂时，一个固定的 RAG 流程不够用。比如：

- 有的问题查产品文档。
- 有的问题查数据库。
- 有的问题查知识图谱。
- 有的问题需要联网搜索。
- 有的问题不需要检索。

Modular RAG 允许系统根据问题类型组合不同模块。

### 优点

- 扩展性强。
- 方便为不同业务定制不同流程。
- 适合和 Agent、工具调用、工作流系统结合。

### 缺点

- 架构复杂。
- 链路变长后，延迟更高。
- 一旦答案错了，排查是“检索错了、重排错了、压缩错了、生成错了”会变得更困难。

### 适合场景

- 企业级 RAG 平台。
- 多业务、多数据源、多权限系统。
- 需要长期演进的 RAG 产品。

### 初学者记忆法

Modular RAG 就是：“把 RAG 做成乐高积木，不同问题拼不同流程。”

## 4. HyDE：假设文档嵌入

### 它是什么

HyDE 的全称是 Hypothetical Document Embeddings。

它的核心想法是：用户的问题有时太短、太口语化，不适合直接做向量检索。那就先让 LLM 写一段“假想答案”或“假想文档”，再用这段更完整的文本去检索。

流程：

```text
用户问题
-> LLM 生成一段假想答案
-> 对假想答案做 Embedding
-> 用假想答案向量检索真实文档
-> LLM 基于真实文档回答
```

例子：

用户问：

> 怎么提速？

这个问题太模糊。HyDE 可能先生成：

> 提升 RAG 系统速度通常可以从缓存、减少检索数量、优化向量数据库索引、使用更小的模型、异步并行检索等方面入手。

然后拿这段更具体的文本去检索。

### 它解决什么问题

它解决“短查询、模糊查询、表达不完整”导致的向量检索效果差。

### 优点

- 对短问题和语义模糊的问题有帮助。
- 实现相对简单。
- 不需要修改文档索引结构。

### 缺点

- 假想答案如果偏了，检索也会被带偏。
- 多了一次 LLM 调用，成本和延迟增加。
- 对事实精确类问题未必有效。

### 适合场景

- 用户问题很短的搜索场景。
- 知识库内容比较语义化，而不是精确关键词匹配。
- 探索式问答。

### 初学者记忆法

HyDE 就是：“先脑补一个可能的答案，再拿这个答案去找真资料。”

## 5. RAG-Fusion：多查询融合检索

### 它是什么

RAG-Fusion 的核心是：不要只用用户原始问题检索，而是让模型生成多个不同角度的查询，然后并行检索，最后融合结果。

流程：

```text
用户问题
-> 生成多个改写查询
-> 每个查询各自检索
-> 用 RRF 等方法融合排序
-> LLM 回答
```

例如用户问：

> GraphRAG 有什么用？

系统可能生成：

- GraphRAG 的核心思想是什么？
- GraphRAG 适合哪些应用场景？
- GraphRAG 相比普通 RAG 的优势是什么？
- GraphRAG 的缺点和成本是什么？

这些查询分别检索后合并，召回会更全面。

### 它解决什么问题

一个问题可能有多种表达方式。只用原始问题检索，容易漏掉相关文档。RAG-Fusion 通过多角度查询提高召回覆盖面。

### 优点

- 召回更全面。
- 对用户表达不清的问题更稳。
- 适合信息分散在多个文档中的场景。

### 缺点

- 检索次数变多，成本和延迟上升。
- 改写查询质量不好时，会引入更多噪声。
- 融合排序策略需要调试。

### 适合场景

- 开放式问题。
- 知识分散的问题。
- 用户问法多样的客服、文档问答、研究助手。

### 初学者记忆法

RAG-Fusion 就是：“同一个问题换几种问法一起搜，再把结果合并。”

---

## 6. Hybrid RAG：混合检索 RAG

### 它是什么

Hybrid RAG 指的是不只依赖向量检索，而是结合多种检索方式：

- 向量检索：适合语义相似。
- BM25/关键词检索：适合精确词、编号、术语。
- SQL 查询：适合结构化数据。
- 图检索：适合实体关系。
- 规则过滤：适合权限、时间、类型限制。

典型流程：

```text
用户问题
-> 向量检索 + 关键词检索 + 结构化过滤
-> 结果融合
-> 重排
-> LLM 回答
```

### 它解决什么问题

纯向量检索不是万能的。比如：

- 查询错误码 `E1027`。
- 查询函数名 `getUserProfile`。
- 查询合同条款编号 `Section 4.2`。
- 查询产品 SKU。

这些场景关键词匹配往往比语义匹配更可靠。

### 优点

- 比纯向量检索稳定。
- 同时兼顾语义相似和精确匹配。
- 对代码、法规、财务、产品文档特别有用。

### 缺点

- 多路结果如何融合比较复杂。
- 不同检索器的分数不可直接比较。
- 系统实现和评测更复杂。

### 适合场景

- 企业知识库。
- 法律、金融、医疗文档。
- 代码搜索。
- 产品文档和工单系统。

### 初学者记忆法

Hybrid RAG 就是：“不要只靠语义搜索，关键词、数据库、图谱都一起用。”

## 7. Rerank RAG：重排序增强 RAG

### 它是什么

Rerank RAG 通常分两步：

1. 先用便宜快速的检索方式召回较多候选，比如 Top-50。
2. 再用更精细的重排模型选出最相关的 Top-5。

流程：

```text
用户问题
-> 初步召回 Top-50
-> Reranker 精排
-> 选择 Top-5
-> LLM 回答
```

### 它解决什么问题

向量数据库召回的 Top-K 不一定排序准确。可能真正有用的文档排在第 12 位，但你只取了前 5 个，它就被漏掉了。

Rerank 的作用是把“粗糙召回”变成“精细选择”。

### 优点

- 通常能显著提升最终上下文质量。
- 对复杂问题和长文档尤其有效。
- 是生产级 RAG 中非常常见的增强。

### 缺点

- 增加一次模型调用。
- 延迟和成本更高。
- 如果初步召回完全没有正确文档，重排也救不了。

### 适合场景

- 需要高准确率的知识库问答。
- 检索结果噪声多的文档集合。
- Top-K 经常包含看似相关但实际无关内容的系统。

### 初学者记忆法

Rerank RAG 就是：“先海选，再复试。”

## 8. Parent-Child RAG：父子块 RAG

### 它是什么

Parent-Child RAG 会把文档切成两种粒度：

- Child chunk：小块，用来做精确匹配。
- Parent chunk：大块，用来提供完整上下文。

流程：

```text
文档 -> 切成大块 Parent -> 每个 Parent 再切小块 Child
用户问题 -> 检索 Child -> 找到对应 Parent -> 把 Parent 给 LLM
```

### 它解决什么问题

如果 chunk 太小，检索很准，但回答时上下文不够。

如果 chunk 太大，上下文完整，但检索容易不准。

Parent-Child RAG 同时利用两者：

- 小块负责“找得准”。
- 大块负责“讲得完整”。

### 优点

- 兼顾精确召回和上下文完整性。
- 对长文档很实用。
- 比单纯调 chunk size 更稳。

### 缺点

- 索引和存储结构更复杂。
- Parent 太大时仍然会引入噪声。
- 需要维护 child 和 parent 的映射关系。

### 适合场景

- 技术手册。
- 法律合同。
- 论文。
- 公司制度文档。

### 初学者记忆法

Parent-Child RAG 就是：“用小段找位置，用大段给上下文。”

## 9. Hierarchical RAG：层级 RAG

### 它是什么

Hierarchical RAG 会保留文档的层级结构，例如：

```text
书 -> 章 -> 节 -> 段落
产品文档 -> 模块 -> 页面 -> 小节
法律文件 -> 编 -> 章 -> 条 -> 款
```

检索时不是直接在所有段落里乱搜，而是先定位大范围，再进入细节。

流程：

```text
用户问题
-> 找相关文档或章节
-> 在相关章节内找具体段落
-> LLM 回答
```

### 它解决什么问题

普通 RAG 把所有 chunk 放在一个平面里，容易丢失文档结构。层级 RAG 利用结构信息，让检索更符合人类阅读方式。

### 优点

- 适合结构清晰的长文档。
- 能减少跨章节误召回。
- 回答时更容易给出来源位置。

### 缺点

- 需要解析和保留文档结构。
- 文档结构混乱时效果会下降。
- 多级检索策略需要设计。

### 适合场景

- 书籍问答。
- 法律法规。
- 标准规范。
- 产品手册。

### 初学者记忆法

Hierarchical RAG 就是：“先找章节，再找段落。”

## 10. RAPTOR：递归摘要树 RAG

### 它是什么

RAPTOR 是一种把文档构造成树状摘要结构的方法。

它会：

1. 把文档切成很多底层 chunk。
2. 对相近 chunk 聚类。
3. 对每组 chunk 生成摘要。
4. 再对摘要继续聚类和摘要。
5. 最后形成一棵从细节到全局的摘要树。

结构大致是：

```text
顶层全局摘要
├── 中层主题摘要
│   ├── 底层原文 chunk
│   └── 底层原文 chunk
└── 中层主题摘要
    ├── 底层原文 chunk
    └── 底层原文 chunk
```

### 它解决什么问题

普通 RAG 擅长回答局部事实问题，但不擅长回答全局问题，比如：

- 这批文档主要讲了什么？
- 这本书的核心观点是什么？
- 多份报告的共同主题是什么？

RAPTOR 通过摘要树，让系统既能检索细节，也能检索高层总结。

### 优点

- 适合全局性问题。
- 能在不同抽象层级检索。
- 对长文档集合很有价值。

### 缺点

- 构建索引成本高。
- 摘要可能丢失细节。
- 摘要如果出错，会影响上层节点。
- 数据更新后需要重新维护摘要树。

### 适合场景

- 长篇书籍。
- 大量研究报告。
- 企业知识库总结。
- 需要主题归纳的场景。

### 初学者记忆法

RAPTOR 就是：“把文档先总结成一棵树，问细节看叶子，问全局看树顶。”

---

## 11. Multi-hop RAG：多跳 RAG

### 它是什么

Multi-hop RAG 用多轮检索来回答复杂问题。它不是一次检索就结束，而是：

```text
问题 -> 检索证据 A -> 推理出中间问题 -> 检索证据 B -> 综合回答
```

例子：

> 写《三体》的作者还写过哪些获得过雨果奖的作品？

这个问题需要多步：

1. 找到《三体》的作者是刘慈欣。
2. 查刘慈欣有哪些作品。
3. 判断哪些作品获得过雨果奖。

### 它解决什么问题

有些问题的答案不在一个文档片段里，需要多个证据链起来。

### 优点

- 适合复杂推理问题。
- 能跨多个文档找证据。
- 比一次性 Top-K 检索更灵活。

### 缺点

- 每一步都可能出错。
- 前一步错了，后面会跟着错。
- 延迟和成本更高。
- 需要设计停止条件，否则可能无限检索。

### 适合场景

- 跨文档问答。
- 研究助手。
- 法律案例分析。
- 复杂企业知识查询。

### 初学者记忆法

Multi-hop RAG 就是：“一个问题分几步查，每查一步再决定下一步。”

---

## 12. Self-RAG：自反思 RAG

### 它是什么

Self-RAG 让模型参与判断：

- 这个问题需不需要检索？
- 检索到的内容有没有用？
- 答案是否被证据支持？
- 是否需要继续检索？

它不是固定地“每个问题都检索 Top-K”，而是让模型在生成过程中进行自我判断。

### 它解决什么问题

固定 RAG 流程有两个问题：

- 简单问题也检索，浪费成本。
- 复杂问题只检索一次，不够可靠。

Self-RAG 试图让系统更自适应。

### 优点

- 更灵活。
- 有机会减少不必要检索。
- 可以在回答前检查证据是否足够。

### 缺点

- 依赖模型自评能力。
- 模型可能错误判断“不需要检索”。
- 实现和训练/提示设计更复杂。
- 生产系统中可控性不如显式规则。

### 适合场景

- 问题类型变化很大的系统。
- 希望模型动态决定检索策略的场景。
- 研究型 RAG 实验。

### 初学者记忆法

Self-RAG 就是：“让模型边答边问自己：我要不要查资料？资料够不够？”

---

## 13. CRAG：纠正式 RAG

### 它是什么

CRAG 的全称常被理解为 Corrective Retrieval-Augmented Generation，即纠正式 RAG。

它的核心思想是：检索结果不能默认可信。系统应该先判断检索结果质量，如果不可靠，就进行纠正。

典型流程：

```text
用户问题
-> 初次检索
-> 评估检索结果质量
-> 如果结果可靠：生成答案
-> 如果结果不可靠：重新检索 / 扩展检索 / Web 搜索 / 过滤错误内容
-> 生成答案
```

### 它解决什么问题

基础 RAG 最大的问题之一是：

> 检索错了，但模型不知道错了，还会一本正经地回答。

CRAG 在生成前加了一个“检索质量检查”。

### 优点

- 能减少基于错误文档生成答案。
- 对开放域问答更稳。
- 可以结合外部搜索补充知识。
- 更符合高可靠系统的设计思路。

### 缺点

- 需要额外的评估器判断检索结果好坏。
- 评估器也可能误判。
- 增加延迟和成本。
- 纠正策略需要设计，比如什么时候重新检索，什么时候联网搜索。

### 适合场景

- 高可靠知识问答。
- 医疗、法律、金融等不希望胡答的领域。
- 知识库覆盖不完整的开放域问答。

### 初学者记忆法

CRAG 就是：“先检查搜到的资料靠不靠谱，不靠谱就重搜或纠正。”

---

## 14. Adaptive RAG：自适应 RAG

### 它是什么

Adaptive RAG 会根据问题难度和类型选择不同策略。

例如：

```text
简单事实问题 -> 直接检索一次
模糊问题 -> 先 Query Rewrite
复杂问题 -> Multi-hop RAG
关系问题 -> GraphRAG
无需外部知识 -> 直接回答
```

它通常需要一个 Router，也就是路由器，先判断问题应该走哪条路径。

### 它解决什么问题

不是所有问题都值得走复杂流程。简单问题用 Agentic RAG 太贵，复杂问题用 Naive RAG 又太弱。

Adaptive RAG 的目标是：不同问题用不同成本的方案。

### 优点

- 成本和效果平衡更好。
- 简单问题响应快。
- 复杂问题可以使用更强流程。
- 适合生产系统。

### 缺点

- 路由器判断错了会影响结果。
- 策略多了之后评测更复杂。
- 需要维护多条 RAG 链路。

### 适合场景

- 用户问题类型差异很大的应用。
- 既关心成本，也关心准确率的系统。
- 多数据源、多工具的企业助手。

### 初学者记忆法

Adaptive RAG 就是：“简单题简单做，难题走高级流程。”

---

## 15. Agentic RAG：智能体式 RAG

### 它是什么

Agentic RAG 让 LLM 不只是最后回答问题，而是参与整个任务过程：

- 分析问题。
- 制定检索计划。
- 选择工具。
- 多轮检索。
- 观察结果。
- 修正计划。
- 最后综合回答。

典型流程：

```text
用户目标
-> Agent 思考需要哪些信息
-> 调用检索工具
-> 阅读结果
-> 决定是否继续查
-> 调用其他工具
-> 生成最终答案
```

### 它解决什么问题

很多真实任务不是一次问答，而是一个小型研究流程。例如：

- 帮我比较 5 篇论文的方法差异。
- 找出代码库里 RAG 相关模块并解释调用链。
- 分析某家公司最近的产品战略。
- 结合数据库、文档和网页给出结论。

这些任务需要规划、检索、判断、再检索。

### 优点

- 能处理复杂任务。
- 可以调用多种工具。
- 能多轮迭代，不局限于一次检索。
- 适合研究助手、数据分析助手、代码助手。

### 缺点

- 延迟高、成本高。
- 行为不如固定流程稳定。
- 容易跑偏或循环。
- 需要工具权限、安全边界、步数限制和日志观测。

### 适合场景

- 研究型问答。
- 代码库分析。
- 多工具工作流。
- 需要自主规划的信息收集任务。

### 初学者记忆法

Agentic RAG 就是：“不是固定搜一次，而是让 Agent 像研究员一样自己查资料。”

---

## 16. GraphRAG：图谱增强 RAG

### 它是什么

GraphRAG 会从文档中抽取：

- 实体：人、公司、地点、产品、概念、事件。
- 关系：谁属于哪个公司，哪个产品依赖哪个模块，哪个事件导致哪个结果。
- 社区：一组密切相关的实体和主题。

然后构建知识图谱，再结合图检索和文本检索回答问题。

简单说，普通 RAG 主要看“文本片段相似度”，GraphRAG 还会看“实体之间的关系”。

### 举例

假设知识库里有很多公司新闻：

```text
Alice 加入 A 公司。
A 公司收购 B 公司。
B 公司开发了产品 X。
产品 X 后来整合进产品 Y。
```

如果用户问：

> Alice 可能间接参与了哪些产品线？

普通向量检索可能找不到完整链路。GraphRAG 可以沿着图关系走：

```text
Alice -> A 公司 -> 收购 B 公司 -> 产品 X -> 产品 Y
```

### 它解决什么问题

GraphRAG 擅长处理：

- 多跳关系。
- 实体网络。
- 全局主题总结。
- 大规模文档的组织结构。
- “谁和谁有什么关系”这类问题。

### 优点

- 比普通 RAG 更擅长关系推理。
- 适合跨文档整合。
- 可以做社区摘要和全局总结。
- 对企业知识网络很有价值。

### 缺点

- 构建知识图谱成本高。
- 实体抽取和关系抽取容易出错。
- 图谱更新维护复杂。
- 对简单问答可能过度设计。
- 图谱质量差时，回答质量也会差。

### 适合场景

- 企业知识管理。
- 人物、组织、项目关系分析。
- 事件链分析。
- 金融风控。
- 情报分析。
- 大量文档的主题归纳。

### 不适合场景

- 小型 FAQ。
- 简单产品文档问答。
- 数据量很少的项目。
- 没有明显实体关系的文本集合。

### 初学者记忆法

GraphRAG 就是：“把文档变成关系网，再沿着关系网找答案。”

---

## 17. Long-context RAG：长上下文 RAG

### 它是什么

Long-context RAG 利用支持超长上下文的模型，把大量候选文档直接放进 Prompt，让模型自己从里面找答案。

流程：

```text
用户问题
-> 粗略检索大量文档
-> 塞进长上下文模型
-> 模型阅读并回答
```

### 它解决什么问题

传统 RAG 的瓶颈是 Top-K 选择。如果选错片段，模型根本看不到正确答案。长上下文模型可以放入更多候选材料，降低漏召回风险。

### 优点

- 工程上相对简单。
- 对召回错误更宽容。
- 适合阅读长报告或大量材料。

### 缺点

- 成本高。
- 延迟高。
- 上下文很长时，模型不一定真的关注关键细节。
- 如果放入大量噪声，答案也可能变差。

### 适合场景

- 文档数量不大，但单篇很长。
- 用户愿意等待更完整的分析。
- 长上下文模型成本可接受。

### 初学者记忆法

Long-context RAG 就是：“少做精细筛选，多给模型材料让它自己读。”

---

## 18. Multimodal RAG：多模态 RAG

### 它是什么

Multimodal RAG 处理的不只是文本，还包括：

- 图片。
- 表格。
- PDF 扫描件。
- 图表。
- 音频。
- 视频。
- 截图。

它可能会用 OCR、图像理解模型、视觉 Embedding、表格解析、语音转文字等技术，把多种信息转成可检索内容。

### 它解决什么问题

很多知识不在纯文本里。例如：

- 论文里的方法图。
- 财报里的表格。
- 医疗影像。
- 工业设备照片。
- 教学视频。
- UI 截图。

普通文本 RAG 看不到这些信息。

### 优点

- 能覆盖真实世界中更丰富的知识。
- 对论文、报告、图表问答很有用。
- 可以回答“这张图说明了什么”这类问题。

### 缺点

- 解析难度高。
- 成本高。
- 多模态检索和评测更复杂。
- 图片或表格解析错了会严重影响答案。

### 适合场景

- 论文阅读助手。
- 财报分析。
- 医疗和工业文档。
- 视频知识库。
- UI/设计资料问答。

### 初学者记忆法

Multimodal RAG 就是：“不只搜文字，也搜图片、表格、音频和视频。”

---

## 19. Cache-Augmented RAG：缓存增强 RAG

### 它是什么

Cache-Augmented RAG 会缓存 RAG 流程中的某些结果：

- 用户问题。
- 改写后的查询。
- 检索结果。
- 重排结果。
- 最终答案。
- Embedding。

### 它解决什么问题

RAG 系统经常有大量重复或相似问题。如果每次都完整检索和生成，会浪费成本。

### 优点

- 降低延迟。
- 降低模型调用成本。
- 高频问题响应更快。
- 对客服和内部知识库很实用。

### 缺点

- 知识库更新后，缓存可能过期。
- 相似问题是否可以复用答案需要谨慎判断。
- 权限相关系统中，缓存必须考虑用户权限隔离。

### 适合场景

- 客服系统。
- 高频 FAQ。
- 企业内部助手。
- 成本敏感的生产环境。

### 初学者记忆法

Cache-Augmented RAG 就是：“问过的问题别每次都重新查、重新答。”

---

## 20. Corrective、Self、Adaptive、Agentic 的区别

这几个名字很容易混淆，可以这样区分：

| 范式 | 核心问题 | 谁来决策 | 重点 |
|---|---|---|---|
| CRAG | 搜到的资料可靠吗？ | 质量评估器 | 检索纠错 |
| Self-RAG | 需不需要检索？证据够不够？ | 模型自己判断 | 自我反思 |
| Adaptive RAG | 这个问题该走哪条流程？ | 路由器 | 策略选择 |
| Agentic RAG | 为完成任务下一步该做什么？ | Agent | 自主规划 |

简单理解：

- CRAG 关心“检索结果错了怎么办”。
- Self-RAG 关心“模型能不能自己判断是否需要查资料”。
- Adaptive RAG 关心“不同问题走不同流程”。
- Agentic RAG 关心“让模型像一个研究员一样多步行动”。

---

## 21. GraphRAG 和普通 RAG 的区别

| 对比项 | 普通 RAG | GraphRAG |
|---|---|---|
| 知识表示 | 文本 chunk | 实体 + 关系 + 文本 |
| 检索依据 | 文本相似度为主 | 图关系 + 文本相似度 |
| 擅长问题 | 局部事实问答 | 多跳关系、全局总结 |
| 构建成本 | 低到中 | 高 |
| 维护难度 | 中 | 高 |
| 适合数据 | 文档、FAQ、手册 | 人物、组织、事件、概念关系密集的数据 |

如果你的问题主要是：

> 这段文档里怎么说？

普通 RAG 往往够用。

如果你的问题主要是：

> A 和 B 之间有什么关系？这些事件背后的共同主题是什么？哪些实体处在同一个关系网络里？

GraphRAG 更有优势。

---

## 22. 初学者应该按什么顺序学习

建议学习顺序如下：

1. **Naive RAG**
   - 先理解最基础的检索增强生成。
   - 重点掌握 chunk、embedding、vector search、prompt 拼接。

2. **Advanced RAG**
   - 学 Query Rewrite、Hybrid Search、Rerank、Metadata Filter。
   - 这是从 Demo 走向可用系统的关键。

3. **HyDE 和 RAG-Fusion**
   - 学会改善用户查询。
   - 理解“问题本身也需要优化”。

4. **Parent-Child RAG 和 Hierarchical RAG**
   - 学会处理长文档。
   - 理解 chunk 粒度和文档结构的重要性。

5. **CRAG 和 Self-RAG**
   - 学会检查检索质量和证据质量。
   - 理解 RAG 的可靠性问题。

6. **Multi-hop RAG 和 Agentic RAG**
   - 学会复杂任务中的多步检索。
   - 理解 RAG 从“问答系统”变成“研究助手”。

7. **GraphRAG 和 RAPTOR**
   - 学会面向大规模知识和全局总结的结构化方法。
   - 理解图谱和摘要树为什么有用。

8. **Multimodal RAG**
   - 最后再处理图片、表格、音频、视频。
   - 这是更复杂的数据理解问题。

---

## 23. 如何选择合适的 RAG 范式

### 如果你只是做入门 Demo

选择：

- Naive RAG。

原因：

- 最简单。
- 能快速跑通完整流程。
- 适合理解基本概念。

### 如果你要做一个真正可用的文档问答系统

选择：

- Advanced RAG。
- Hybrid RAG。
- Rerank RAG。

原因：

- 这是生产环境最常见组合。
- 成本和效果比较平衡。

### 如果你的用户问题很短、很模糊

选择：

- HyDE。
- RAG-Fusion。
- Query Rewrite。

原因：

- 先把问题变得更适合检索。

### 如果你的文档很长

选择：

- Parent-Child RAG。
- Hierarchical RAG。
- RAPTOR。
- Long-context RAG。

原因：

- 长文档不能只靠简单切块。

### 如果你的问题需要多个证据

选择：

- Multi-hop RAG。
- Agentic RAG。
- GraphRAG。

原因：

- 单次检索很难完成复杂推理。

### 如果你非常在意答案可靠性

选择：

- CRAG。
- Self-RAG。
- Rerank RAG。
- 答案验证模块。

原因：

- 需要检查检索结果和最终答案是否被证据支持。

### 如果你的知识中实体关系很多

选择：

- GraphRAG。

原因：

- 图结构比纯文本 chunk 更适合关系推理。

---

## 24. 一个推荐的生产级组合

很多实际系统可以从这个组合开始：

```text
用户问题
-> 问题分类 / 路由
-> Query Rewrite
-> Hybrid Search
-> Rerank
-> Parent-Child Context Expansion
-> 答案生成
-> 引用来源
-> 答案校验
```

如果还不够，再逐步加入：

- CRAG：检索结果不可靠时纠错。
- RAG-Fusion：召回不全时多查询融合。
- Agentic RAG：任务需要多步规划时使用。
- GraphRAG：知识关系网络非常重要时使用。
- RAPTOR：需要全局总结大量长文档时使用。

## 25. 常见误区

### 误区一：RAG 就是向量数据库

不是。向量数据库只是 RAG 的一个组件。真正影响效果的还有数据清洗、chunking、检索策略、重排、上下文组装、评测和安全。

### 误区二：Embedding 模型越强，RAG 就越好

不一定。Embedding 很重要，但如果文档切分差、数据脏、检索策略错，强 Embedding 也救不了。

### 误区三：Top-K 越大越好

不一定。Top-K 太小会漏信息，太大会引入噪声，还会浪费 token。

### 误区四：GraphRAG 一定比普通 RAG 高级

不一定。GraphRAG 适合关系密集、需要全局理解的场景。简单 FAQ 用 GraphRAG 可能是过度设计。

### 误区五：Agentic RAG 一定最强

不一定。Agentic RAG 适合复杂任务，但它更慢、更贵、更难控。简单问答用固定流程更稳定。

### 误区六：长上下文可以替代 RAG

不完全。长上下文能缓解召回问题，但成本高，而且模型在长上下文中仍可能忽略关键细节。好的检索仍然重要。

---

## 26. 最后总结

如果只记住几句话：

- **Naive RAG**：最基础，先搜再答。
- **Advanced RAG**：生产常用，加强检索和上下文质量。
- **HyDE**：先生成假想答案，再拿它检索。
- **RAG-Fusion**：一个问题改成多个问法一起搜。
- **Hybrid RAG**：向量、关键词、数据库、图谱一起用。
- **Parent-Child RAG**：小块负责找，大块负责答。
- **RAPTOR**：把文档总结成树，兼顾全局和细节。
- **Multi-hop RAG**：复杂问题分多步查。
- **CRAG**：检查检索结果，不靠谱就纠正。
- **Self-RAG**：模型自己判断要不要检索、证据够不够。
- **Adaptive RAG**：简单问题简单做，复杂问题走复杂流程。
- **Agentic RAG**：让 Agent 自主规划和多轮检索。
- **GraphRAG**：把知识变成关系网，适合关系推理和全局总结。
- **Multimodal RAG**：不只处理文本，也处理图片、表格、音视频。

学习 RAG 时，最推荐的路径不是一开始就追 GraphRAG 或 Agentic RAG，而是先把基础 RAG 做扎实，再逐步解决真实问题：

```text
检索不准 -> Query Rewrite / Hybrid Search / Rerank
上下文不够 -> Parent-Child / Hierarchical RAG
问题复杂 -> Multi-hop / Agentic RAG
资料不可靠 -> CRAG / Self-RAG
关系很多 -> GraphRAG
文档太长 -> RAPTOR / Long-context RAG
数据不止文本 -> Multimodal RAG
```
