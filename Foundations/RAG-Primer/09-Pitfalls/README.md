# 09 · 常见踩坑速查（Pitfalls）

把前面 8 个模块里**容易翻车的点**集中收录。建议在排查 RAG 系统问题时
先按本清单过一遍，再去做更复杂的实验。

---

## 一句话理解

> 大多数 RAG "调不好"的项目，问题不在前沿技术，而在**几个低级错误叠加**。

---

## Top 10 翻车清单

### 1. PDF 加载就坏了
**症状**：召回总是奇怪段落、表格里数字乱跳。
**原因**：双栏 PDF 串行、表格被打散、页眉页脚混入。
**解法**：先看 [`../01-Loading/`](../01-Loading/)，验证一份原始解析结果。

### 2. chunk 太大，语义被稀释
**症状**：召回 top-1 包含答案，但 LLM 答非所问。
**原因**：chunk 800+ token 时，关键句被周围噪声淹没。
**解法**：缩小 chunk 至 200–400，或上 Parent-Child。

### 3. chunk 太小，缺上下文
**症状**：召回片段无法独立理解（"这" "他" 指代不明）。
**解法**：加 overlap、上 Parent-Child、加 contextual chunking。

### 4. embedding 没归一化
**症状**：相似度结果"长向量赢"，排序奇怪。
**解法**：所有 embedding 入库前 L2 归一化，统一用 cosine。

### 5. 阈值设死
**症状**：很多 query 召回为 0 或召回过多噪声。
**原因**：用 `score > 0.7` 作硬阈值，忽视分布漂移。
**解法**：取 top-K 而不是阈值；如需阈值，按业务集校准。

### 6. 中文 BM25 没分词
**症状**：BM25 在中文语料几乎没贡献。
**原因**：`split()` 按空格切，中文一个 token 都切不出。
**解法**：用 jieba / pkuseg 先分词，或换基于字 n-gram 的方案。

### 7. 用了 reranker 但召回 K 太小
**症状**：上了 reranker 没明显提升。
**原因**：召回 top-3 → reranker，重排空间太小。
**解法**：召回 top-30 ~ 100，再 rerank 到 top-5。

### 8. Lost in the Middle
**症状**：上下文给了 20 段，答案在中间却答不出。
**原因**：LLM 对中间位置上下文注意力低。
**解法**：reranker 把最相关放前后；缩短上下文条数。

### 9. 评估在玩自我游戏
**症状**：每次"优化"都涨点，但用户体感无变化。
**原因**：dev 集天天调，没有冻结 test。
**解法**：固定 test 集，调优只在 dev 上做。

### 10. 召回好但答错，怪 LLM
**症状**：明明召回里有答案，LLM 还是答错或幻觉。
**原因**：prompt 没强调"基于上下文"，或上下文与问题对不齐。
**解法**：检查 RAGAS Faithfulness；改 prompt 用 cite 模式。

---

## 排查流程图

```
LLM 答案错？
   │
   ├─ Faithfulness 低？  → prompt / 上下文格式问题
   │
   ├─ Context Recall 低？→ 检索阶段（embedding / chunk / 混合）
   │
   ├─ Context Precision 低？→ 加 reranker / 缩小 K
   │
   └─ Answer Relevancy 低？→ query 改写 / 多轮澄清
```
