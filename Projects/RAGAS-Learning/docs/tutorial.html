01 环境搭建与 import 验证

SETUP PASSING

安装 RAGAS 及其依赖，确保 import ragas 成功并输出版本号。这是所有后续章节的基础。

1
确认 Python 版本与虚拟环境 ▶

本仓库使用 uv 管理虚拟环境。Python 版本需要 >= 3.12。

# 检查 Python 版本
.venv/bin/python --version
# Python 3.12.x

# 如果没有虚拟环境，先创建
uv venv
uv sync
📚
为什么用 uv？ uv 是 Rust 编写的 Python 包管理器，比 pip 快 10-100 倍。它用 pyproject.toml 锁定依赖，确保跨会话一致性。
2
安装 ragas 和 datasets ▶

在仓库根目录运行：

# 添加 ragas 和 datasets 到项目依赖
uv add ragas datasets

安装完成后 pyproject.toml 会自动更新：

"datasets>=4.8.5",
"ragas>=0.4.3",
📚
RAGAS 依赖了什么？ ragas 核心依赖：langchain（LLM 调用）、datasets（数据集格式）、instructor（结构化输出）、tiktoken（token 计数）。约 276 个子依赖。
3
解决依赖冲突（关键） ▶

安装后直接 import ragas 可能报错。实测遇到的冲突：

# 报错信息
ImportError: cannot import name 'Mistral' from 'mistralai'

根因：instructor 1.15.1 在 providers/mistral/client.py 中调用 from mistralai import Mistral，但 mistralai 2.x 已经移除了这个类名。

修复：在 pyproject.toml 中锁定 mistralai 版本：

"mistralai<2",
"ragas>=0.4.3",
# 然后重新同步
uv sync
⚠️
重要：不锁定 mistralai 版本的话，uv sync 每次都会还原到 2.x，导致 import 再次失败。必须写入 pyproject.toml 才能持久化。
4
验证 import 成功 ▶ 
import ragas
print(ragas.__version__)
# 0.4.3

也可以直接运行项目初始化脚本：

./init.sh
==> 虚拟环境已激活 (.venv) ==> 同步依赖 (uv sync) ==> 验证 RAGAS 导入 OK RAGAS (0.4.3) OK HuggingFace Datasets (4.8.5) OK LangChain (1.2.10) 全部 3 个依赖就绪 ==> 初始化完成
✅
ragas-001 验证通过！ RAGAS 0.4.3 + Datasets 4.8.5 + LangChain 1.2.10 全部就绪，可以开始学习核心指标了。
5
修复 init.sh 的 Bash/Python 兼容问题 ▶

init.sh 内嵌 Python 代码验证依赖。原始写法用双引号包裹 Python 代码，导致 bash 展开 f-string 中的 ${...}。

# 错误写法 — bash 会展开 ${ver}
python -c "
print(f'OK {label} (${ver})')   # bash 把 ${ver} 当变量展开
"

# 正确写法 — 使用单引号 heredoc，bash 不做变量展开
python << 'PYEOF'
print(f'OK {label} ({ver})')   # Python 正确处理 f-string
PYEOF
📚
Shell heredoc 小知识： << 'EOF'（带引号）禁止变量展开，<< EOF（无引号）允许展开。嵌入 Python 时始终用带引号的版本。

02 Faithfulness（忠实度）

METRICS PASSING

Faithfulness 衡量模型生成的答案是否忠实于检索到的上下文，是检测幻觉（Hallucination）的核心指标。

1
RAGAS 0.4.3 API 调用方式 ▶

RAGAS 0.4.x 使用 SingleTurnSample 封装输入，通过 single_turn_ascore() 异步评估：

from ragas.dataset_schema import SingleTurnSample
from ragas.metrics._faithfulness import Faithfulness

sample = SingleTurnSample(
    user_input="问题",
    response="答案",
    retrieved_contexts=["上下文1"],
)
scorer = Faithfulness(llm=llm)
score = await scorer.single_turn_ascore(sample)
⚠️
必须显式传入 LLM：RAGAS 0.4.3 不会自动从环境变量创建 LLM。需要 llm_factory(model, client=OpenAI(...)) 手动创建。
2
选择评估用 LLM（关键） ▶

RAGAS 通过 instructor 库要求 LLM 返回纯净 JSON。推理模型（带思考过程）会导致解析失败。

⚠️
不可用：MiniMax-M2/M2.7、GLM-5.1 等推理模型。思考标签破坏 JSON 解析。
可用：glm-4-flash、glm-4-plus、gpt-4o-mini 等非推理模型。
from openai import OpenAI
from ragas.llms import llm_factory

client = OpenAI(
    api_key="your-key",
    base_url="https://open.bigmodel.cn/api/paas/v4",
)
llm = llm_factory("glm-4-flash", client=client)
3
运行结果 ▶
高忠实度（答案基于上下文）: 1.0000 低忠实度（包含幻觉）: 0.0000 验证通过: 高忠实度得分 > 低忠实度得分
✅
ragas-002 验证通过！基于上下文的答案得分 1.0，编造幻觉的答案得分 0.0。

03 Answer Relevancy（答案相关性）

METRICS PASSING

Answer Relevancy 衡量生成的答案与用户提问的相关程度。答非所问会得到低分。

1
核心原理 ▶

Answer Relevancy 的评估方式很巧妙：它不直接比较问题和答案，而是从答案反向生成可能的问题，然后计算生成问题与原始问题的语义相似度。

📚
因此它需要两个组件：LLM（生成反向问题）+ Embeddings（计算语义相似度）。
2
完整源码 ▶ 
import asyncio
from dotenv import load_dotenv
from openai import OpenAI
from langchain_openai import OpenAIEmbeddings
from ragas.dataset_schema import SingleTurnSample
from ragas.embeddings import LangchainEmbeddingsWrapper
from ragas.llms import llm_factory
from ragas.metrics._answer_relevance import ResponseRelevancy

load_dotenv("../../.env")

# LLM — 用 glm-4-plus（flash 的反向问题质量不足）
client = OpenAI(
    api_key=os.environ["ZHIPU_API_KEY"],
    base_url="https://open.bigmodel.cn/api/paas/v4",
)
llm = llm_factory("glm-4-plus", client=client)

# Embeddings — 智谱 embedding-3
emb = OpenAIEmbeddings(
    model="embedding-3",
    api_key=os.environ["ZHIPU_API_KEY"],
    base_url="https://open.bigmodel.cn/api/paas/v4",
)
embeddings = LangchainEmbeddingsWrapper(embeddings=emb)

# 评估
scorer = ResponseRelevancy(llm=llm, embeddings=embeddings, strictness=1)
sample = SingleTurnSample(
    user_input="Python 的列表推导式是什么？",
    response="列表推导式是从已有列表创建新列表的简洁语法...",
)
score = await scorer.single_turn_ascore(sample)
3
运行结果 ▶
相关答案（切题）: 0.8739 不相关答案（答非所问）: 0.4567 验证通过: 相关答案得分 > 不相关答案
📚
注意：分数不是 1.0 vs 0.0，因为语义相似度是连续值。0.87 已经是很高的相关性，0.46 表示明显不相关但仍有语义重叠。

04 Context Precision（上下文精度）

METRICS PASSING

Context Precision 衡量检索到的上下文中，相关内容是否排名靠前。好的检索系统应该把最相关的片段排在前面。

1
核心原理 ▶

检索系统返回的上下文是有序的。Context Precision 关注：相关上下文是否排在前面？

LLM 逐个判断每个上下文是否与问题相关，然后根据排名位置计算加权分数。排在前面的相关上下文权重更高。

📚
这个指标只需要 LLM，不需要 Embeddings。所需的输入列：user_input + reference（标准答案）+ retrieved_contexts。
2
完整源码 ▶ 
from ragas.dataset_schema import SingleTurnSample
from ragas.metrics._context_precision import ContextPrecision

# 相关上下文排第一 → 高精度
sample_high = SingleTurnSample(
    user_input="相对论是谁提出的？",
    reference="相对论由爱因斯坦提出。",
    retrieved_contexts=[
        "爱因斯坦于1905年提出狭义相对论...",  # ✓ 相关，排第一
        "牛顿经典力学在高速下不适用...",       # ✗ 不相关
        "量子力学研究微观粒子行为...",           # ✗ 不相关
    ],
)
scorer = ContextPrecision(llm=llm)
score = await scorer.single_turn_ascore(sample_high)
# score = 1.0000
3
运行结果 ▶
高精度（相关排第一）: 1.0000 低精度（相关排最后）: 0.3333 验证通过: 相关上下文排前面的用例得分更高
✅
ragas-004 验证通过！相关排第一得分 1.0，排最后得分 0.33（1/3）。完美体现排名对分数的影响。

05 Context Recall（上下文召回）

METRICS PASSING

Context Recall 衡量标准答案（ground truth）中的信息是否都被检索到的上下文覆盖。遗漏关键信息会扣分。

1
核心原理 ▶

Context Recall 的视角和 Precision 相反：它不关心检索到了多少不相关的，而是关心该检索到的都检索到了吗？

LLM 将 reference 拆分为多条陈述，然后逐条判断每条陈述是否能从上下文中找到支持。覆盖率 = 被支持的陈述数 / 总陈述数。

📚
Precision vs Recall：
Precision = 检索到的有多准？（关注误检）
Recall = 该检索到的都检索到了吗？（关注漏检）
2
完整源码 ▶ 
from ragas.metrics._context_recall import ContextRecall

sample = SingleTurnSample(
    user_input="爱因斯坦有哪些重要贡献？",
    reference="爱因斯坦提出了狭义和广义相对论，"
              "并因光电效应获得诺贝尔奖。",
    retrieved_contexts=[
        "爱因斯坦于1905年提出狭义相对论...",
        "爱因斯坦因光电效应获得1921年诺贝尔奖。",
    ],
)
scorer = ContextRecall(llm=llm)
score = await scorer.single_turn_ascore(sample)
# score = 1.0（全部覆盖）
3
运行结果 ▶
高召回（完整覆盖）: 1.0000 低召回（遗漏诺贝尔奖）: 0.5000 验证通过: 完整召回的用例得分更高
✅
ragas-005 验证通过！完整覆盖得分 1.0。遗漏诺贝尔奖信息时，reference 中 2 条陈述只有 1 条被覆盖，得分 0.5。
